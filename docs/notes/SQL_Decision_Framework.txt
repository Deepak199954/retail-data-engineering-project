# SQL Concepts

---

# SELECT

## Purpose

Retrieve required columns from a table.

## Use When

- Fetching data.
- Creating reports.

## Syntax

```sql
SELECT column1, column2
FROM table_name;
```

---

# WHERE

## Purpose

Filter rows before aggregation.

## Use When

Need only matching rows.

## Syntax

```sql
SELECT *
FROM orders
WHERE order_amount > 1000;
```

---

# GROUP BY

## Purpose

Summarize data.

## Use When

Business wants totals, counts or averages.

## Examples

- Sales by Customer
- Revenue by City
- Orders by Category

## Syntax

```sql
SELECT
    customer_id,
    SUM(order_amount)
FROM orders
GROUP BY customer_id;
```

## Don't Use

When every row must remain in output.

Use Window Functions instead.

---

# HAVING

## Purpose

Filter aggregated results.

## Syntax

```sql
SELECT
    customer_id,
    SUM(order_amount)
FROM orders
GROUP BY customer_id
HAVING SUM(order_amount) > 5000;
```

## Difference

WHERE → filters rows

HAVING → filters groups

---

# CASE

## Purpose

Business conditional logic.

## Syntax

```sql
CASE
WHEN order_amount > 1000 THEN 'High'
ELSE 'Low'
END
```

## Examples

Customer Segmentation

Order Status

Priority

---

# INNER JOIN

## Purpose

Return only matching rows.

## Syntax

```sql
SELECT *
FROM orders o
INNER JOIN customers c
ON o.customer_id=c.customer_id;
```

---

# LEFT JOIN

## Purpose

Keep every row from left table.

## Use When

Missing relationships are allowed.

---

# FULL OUTER JOIN

## Purpose

Keep all rows from both tables.

---

# CROSS JOIN

## Purpose

Generate every possible combination.

## Use When

No relationship exists.

Examples

- Products × Stores
- Calendar × Stores
- Cities × Categories

## Avoid

Fact Table × Fact Table

---

# UNION

## Purpose

Append two datasets.

Duplicates removed.

---

# UNION ALL

## Purpose

Append datasets.

Duplicates preserved.

Preferred for performance.

---

# CTE

## Purpose

Improve readability.

Create intermediate business results.

## Syntax

```sql
WITH customer_sales AS
(
SELECT ...
)
SELECT *
FROM customer_sales;
```

---

# Scalar Subquery

## Purpose

Return one value.

## Examples

Maximum Salary

---

# IN

## Purpose

Compare against multiple values.

## Example

```sql
WHERE city IN ('Pune','Mumbai')
```

---

# Correlated Subquery

## Purpose

Evaluate each outer row individually.

## Use When

---

# EXISTS

## Purpose

Check whether related data exists.

## Returns

TRUE / FALSE

Efficient for existence checks.

# ROW_NUMBER()

## Purpose

Return exactly one row.

## Use

## Syntax

```sql
ROW_NUMBER() OVER(
PARTITION BY customer_id
ORDER BY order_date
)
```

# Window Frame

## Default

```sql
UNBOUNDED PRECEDING
TO CURRENT ROW
```

Best for Running Calculations.

---

## Explicit

```sql
ROWS BETWEEN
UNBOUNDED PRECEDING
AND UNBOUNDED FOLLOWING
```

Used for

LAST_VALUE

Overall MIN/MAX

Overall Aggregates

# NULL Handling

## Check NULL

```sql
WHERE city IS NULL
```

---

## Check NOT NULL

```sql
WHERE city IS NOT NULL
```

---

## String "NULL"

```sql
WHERE city='NULL'
```

Looks for text "NULL"

Not SQL NULL.

---

## Never Use

```sql
WHERE city=NULL
```

Returns no rows because NULL cannot be compared using =.

---

# DISTINCT

## Purpose

Return unique values.

## Internally

Reads complete dataset.

Uses Hashing or Sorting.

Can be expensive on large datasets.

# Common Interview Mistakes

❌ Using GROUP BY when every row is required.

✔ Use Window Functions.

---

❌ Using ROW_NUMBER() when ties matter.

✔ Use RANK().

---

❌ Using RANK() when exactly one row is needed.

✔ Use ROW_NUMBER().

---

❌ Using WHERE after GROUP BY.

✔ Use HAVING.

---

❌ Comparing NULL using =

✔ Use IS NULL

---

❌ CROSS JOIN on transactional tables.

✔ CROSS JOIN dimension tables.

---

# Spark Notes

GROUP BY

- Shuffle Required
- Expensive

---

DISTINCT

- Reads complete dataset
- Shuffle Required

---

ORDER BY

- Global Sort
- Expensive

---

Window Functions

- Partition + Sort
- Usually Shuffle Required

# Final Learning Principle

> **Never choose SQL syntax first.**

Always think in this order:

```
Business Requirement
        ↓
Business Entity
        ↓
Output Grain
        ↓
Column Ownership
        ↓
Requirement Validation
        ↓
Choose SQL Concept
        ↓
Write SQL
        ↓
Review
        ↓
Production Discussion
```

This approach emphasizes solving business problems rather than memorizing SQL syntax and forms the foundation for the Bronze → Silver → Gold Data Engineering pipeline.
